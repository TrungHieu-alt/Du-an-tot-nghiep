\section{Cài đặt backend runtime}

App runtime bắt đầu từ \texttt{backend/src/jobconnect/main.py} và \texttt{backend/src/jobconnect/app.py}. App mount production API routers dưới base prefix \texttt{/api}. Health route \texttt{/api/health} và root \texttt{/} là runtime-only surfaces. Domain router aggregator nằm ở \texttt{backend/src/jobconnect/modules/api/router.py}; file này import các router domain, xây dựng \texttt{ALL\_API\_ROUTERS} và re-export compatibility symbols phục vụ tests/entrypoints.

\section{Cài đặt cơ sở dữ liệu}

Migration production MVP nằm tại \texttt{backend/db/migrations/001\_production\_mvp.sql}. Theo README, data không auto-load chỉ bằng \texttt{docker compose up}; cần chạy \texttt{docker compose exec backend python db/apply\_migrations.py}. Volume \texttt{postgres\_data} giữ dữ liệu qua \texttt{down/up}; chỉ \texttt{down -v} mới xóa volume. Điều này ảnh hưởng đến việc thực nghiệm: nếu test thủ công trên database đã có dữ liệu, cần biết dữ liệu có thể tồn tại từ lần chạy trước.

Schema production dùng PostgreSQL + pgvector. Các bảng embedding dùng \texttt{VECTOR(384)}. Mỗi bảng quan trọng có index tương ứng: users theo role/status, resumes theo owner/status và public pool, jobs theo organization/recruiter/status/published, GIN index cho skills/certifications, parse jobs theo status/created, applications theo job/resume/candidate/status và audit logs theo actor/target/event type \cite{lld-schema}.

\section{Cài đặt document upload và parse worker}

Theo implementation matrix, \texttt{POST /api/documents} đã chuyển sang multipart upload, stream file qua Storage adapter, insert \texttt{uploaded\_documents}, insert queued \texttt{parse\_jobs} và write audit event. Default storage backend là local filesystem dưới \texttt{STORAGE\_LOCAL\_ROOT}; future cloud adapter chỉ cần implement cùng protocol \cite{implementation-matrix}. Endpoint reject unsupported MIME với \texttt{415} và oversize file với \texttt{413}.

Document detail endpoint đã include linked parse jobs, và download URL endpoint trả qua Storage adapter. Local adapter trả \texttt{local://<object\_key>}; future cloud adapter sẽ trả presigned HTTPS URL. Retry parse job tạo queued parse job mới và audit \texttt{parse\_job\_retried}.

Tuy nhiên implementation matrix cũng ghi rõ parse job response hiện vẫn thin so với target. Target review payload cần normalized fields, hard-filter fields, extracted text/reference, parser metadata, embedding metadata, warnings và linked \texttt{resume\_id}/\texttt{job\_id}. Đây là khoảng cách quan trọng với frontend parse review page. Vì vậy báo cáo đánh giá phần document pipeline là ``foundation aligned, review payload partial'' chứ không tuyên bố đã hoàn chỉnh toàn bộ ingestion target.

\section{Cài đặt matching và semantic search}

Runtime matching hiện có hai endpoint chính: \texttt{POST /api/matching/jobs/\{job\_id\}/run} cho recruiter/admin và \texttt{POST /api/matching/resumes/\{resume\_id\}/run} cho candidate/admin. Cả hai endpoint đọc anchor, validate ownership và visibility, lấy opposite pool, apply hard filters, score deterministic, attempt local rerank và trả warning metadata nếu rerank fallback \cite{implementation-matrix}.

Matching request có \texttt{top\_k} default 10, \texttt{min\_score} default 0.7 và \texttt{rerank} tương thích. Runtime Slice 8 ghi nhận rerank được attempted regardless request flag và fallback gracefully trên provider failure. Điều này có nghĩa API không phụ thuộc tuyệt đối vào reranker; nếu sentence-transformers model không load được hoặc lỗi, kết quả deterministic vẫn trả về.

\begin{table}[H]
\caption{Các bước matching trong runtime}
\centering
\begin{tabularx}{\textwidth}{p{0.24\textwidth}X}
\toprule
\textbf{Bước} & \textbf{Cài đặt/ý nghĩa}\\
\midrule
Anchor validation & Published job cho job anchor; active resume cho resume anchor; ownership theo role.\\
Opposite pool & Active resumes hoặc published jobs tùy chiều matching.\\
Hard filters & Job type/location/seniority/education/certifications; missing data fail pair.\\
Scoring & Công thức deterministic với title, skills, req-exp và req-summary components.\\
Rerank & Local cross-encoder trên top window; fallback warning nếu unavailable.\\

\bottomrule
\end{tabularx}
\end{table}

Semantic search cũng đã có endpoints \texttt{/api/candidate/resumes/semantic-search} và \texttt{/api/jobs/semantic-search}. Resume semantic search compute query embedding, rank theo CV summary/experience embeddings; job semantic search rank theo JD requirement embedding. Provider embedding failures trả \texttt{503 embedding\_unavailable} envelope ở các route semantic search đã hardened trong Slice 8 \cite{implementation-matrix}. Đây là một điểm cài đặt cụ thể cho thấy hệ thống đã xử lý failure provider ở bề mặt API, không chỉ ghi trong thiết kế.

\section{Cài đặt auth, authorization và lifecycle}

Auth runtime có \texttt{POST /api/auth/register}, \texttt{POST /api/auth/login}, \texttt{POST /api/auth/logout} và \texttt{GET /api/me}. Implementation matrix ghi nhận register/login tạo hoặc đọc user, hash password, issue JWT với \texttt{expires\_in}; logout stateless trả \texttt{204}; \texttt{/api/me} trả user và role-specific bootstrap profile/organization \cite{implementation-matrix}.

Authorization Slice 2 đã xử lý các điểm quan trọng: admin không đọc candidate/recruiter profile qua endpoint own-profile nữa; organization update kiểm tra recruiter membership; job matching yêu cầu recruiter ownership; resume matching yêu cầu candidate ownership; disabled users bị reject bởi \texttt{require\_active}. Đây là các điểm cần nhấn mạnh vì nếu thiếu ownership check, matching có thể leak dữ liệu hoặc cho phép user thao tác sai tài nguyên.

Lifecycle Slice 3 và Slice 9 đã bổ sung partial PATCH, transition guard cho job publish/close, resume activate/archive, application status graph và terminal-state lock. Invite creation reject duplicate pending invite với \texttt{409 duplicate\_invite}; invite accept trả explicit response \texttt{\{ invite, application \}}; closed-job blocks còn được tracked như follow-up trong Slice 9 matrix.

\section{Cài đặt notifications và audit}

Notifications runtime có list, read và read-all. \texttt{POST /api/notifications/read-all} trả \texttt{updated\_count}; đây là breaking fix so với key cũ \texttt{updated} trong matrix. Application apply/status update và invite send/accept/reject tạo notification rows. Tuy nhiên email provider chưa được gọi; notification rows dùng \texttt{email\_delivery\_status = queued}. Target yêu cầu email delivery failure không rollback business transaction, nên phần email adapter là hướng hoàn thiện tiếp theo \cite{hld-security,implementation-matrix}.

Audit runtime được dùng cho nhiều lifecycle actions: organization create/update, resume activate/archive, job publish/close, document upload/retry, application apply/status, invite sent/accepted/rejected và một số admin monitoring access. Matrix ghi nhận audit consistency vẫn là medium/low priority gap: một số admin reads audit access, một số chưa. Điều này không làm hỏng core workflow, nhưng ảnh hưởng đến observability và governance khi hệ thống chuyển production.

\section{Đánh giá theo yêu cầu sản phẩm}

\subsection{Điểm đáp ứng tốt}

Thứ nhất, shared-pool marketplace được thể hiện rõ bằng visibility rule: active resumes và published jobs. Cả schema, HLD và runtime docs đều nhất quán với quy tắc này. Thứ hai, matching không tạo application; apply hoặc accepted invite mới tạo application. Điều này bảo vệ ranh giới giữa recommendation và business action. Thứ ba, API namespaces production đã thay thế prototype routes, giúp frontend tương lai tích hợp theo \texttt{/api/*}. Thứ tư, provider boundaries cho LLM, embedding, storage và rerank giúp hệ thống chạy được local đồng thời sẵn sàng dùng service bên ngoài.

\subsection{Điểm còn partial}

Parse review payload là gap lớn nhất đối với trải nghiệm upload-parse-review. Target yêu cầu parse job detail expose normalized fields, hard-filter fields, extracted text/reference, parser metadata, embedding metadata, warnings và linked target ID. Runtime hiện đã có parse jobs nhưng response còn thin. Nếu frontend review screen cần highlight field confidence hoặc cho user sửa từng field, API phải mở rộng.

Email delivery cũng còn partial. Business notifications đã có in-app rows, nhưng email provider chưa được gọi. Nếu sản phẩm cần gửi email invite/application status, hệ thống cần email adapter, retry policy, delivery failure logging và possibly dead-letter/ops monitoring.

Application/Invite DTO còn thiếu linked display summaries ở một số list/detail. Đây là gap frontend integration: nếu table cần job title, organization name hoặc resume title mà response chưa denormalize, frontend sẽ phải fetch thêm nhiều endpoint, tăng latency và complexity.

\subsection{Rủi ro kỹ thuật}

Rủi ro thứ nhất là chất lượng parser/embedding khi chuyển từ local fallback sang provider thật. Local deterministic parser ổn cho dev/test nhưng không đại diện hoàn toàn cho CV/JD thực tế. OpenAI-compatible parser cần kiểm soát enum validation và failure modes. Rủi ro thứ hai là thiếu labeled evaluation data nên không thể kết luận production matching quality. Tài liệu yêu cầu đã nói rõ production quality benchmarking conclusions không thuộc MVP nếu chưa có labeled evaluation data \cite{requirements}. Rủi ro thứ ba là frontend chưa active runtime, nên các giả định về response shape cần được khóa qua OpenAPI trước khi triển khai UI.

\section{Kịch bản thực nghiệm đề xuất}

Vì repository hiện có unit tests và docs nhưng chưa có labeled benchmark dataset, phần thực nghiệm phù hợp nhất là smoke-contract và scenario-based verification. Các kịch bản đề xuất gồm:

\begin{enumerate}
  \item Auth smoke: register candidate/recruiter, login, gọi \texttt{/api/me}, kiểm tra JWT exp và role bootstrap.
  \item Candidate flow: create candidate profile, create draft resume, activate resume, verify resume xuất hiện trong recruiter search.
  \item Recruiter flow: create organization/profile, create draft job, publish job, verify job xuất hiện trong candidate search.
  \item Document flow: upload PDF, verify document row, parse job queued/processing/succeeded hoặc failed có error.
  \item Matching flow: tạo một published job và một active resume pass hard filters, run matching, kiểm tra final score, component scores và reasoning.
  \item Application flow: candidate apply, verify application submitted, application event, notification và audit.
  \item Invite flow: recruiter invite, candidate accept, verify application created hoặc reused; reject invite không tạo application.
  \item Admin flow: admin list users/documents/parse jobs/audit logs với pagination/filter.
\end{enumerate}

Các kịch bản này bám sát target runtime flows, không cần dataset lớn nhưng đủ chứng minh hệ thống hoạt động xuyên suốt. Khi có labeled data, có thể bổ sung evaluation metrics cho ranking như precision@k, recall@k, nDCG hoặc agreement với reviewer. Tuy nhiên nếu chưa có nhãn, báo cáo chỉ nên nói về correctness/smoke/contract, không nên claim chất lượng matching ở mức production.

\section{Nhận xét chương}

Chương này cho thấy JobConnect đã có nền tảng implementation tương đối đầy đủ cho backend MVP: FastAPI route surface, PostgreSQL schema, documents/upload foundation, matching helpers, lifecycle modules, notifications và admin monitoring. Các dependency và file test cụ thể xác nhận hệ thống không chỉ là thiết kế trên giấy. Đồng thời, implementation matrix chỉ ra rõ các phần còn partial, đặc biệt parse review payload, email provider, frontend display DTOs và benchmark data. Đây là cơ sở để phần Kết luận đưa ra hướng phát triển thực tế thay vì liệt kê chung chung.
