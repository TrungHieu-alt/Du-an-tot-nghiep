# Backend HLD V2: Matching Run-Only Pipeline

## Goal

Kiểm tra nhanh matching method trên dữ liệu đã có trong PostgreSQL bằng cách nhập `job_id` hoặc `cv_id` và nhận ranked matches có điểm + reasoning.

## Pipeline Stages

1. Load anchor + candidate pool
- Input là `job_id` hoặc `cv_id`.
- Đọc record + embeddings hiện có trong PostgreSQL.
- Embeddings are local MiniLM 384-dimensional vectors when generated by runtime
  tooling; no remote AI/embedding API is called.
- Prototype dùng exhaustive candidate scoring trên dataset test; chưa cần ANN/index tuning.

2. Hard filters
- Hard filter áp dụng hai chiều trên trường chung giữa JD và CV.
- `job_type` chỉ nhận `remote|fulltime|parttime`.
- Nếu JD `job_type=remote` thì bỏ qua hard filter `location`.
- Nếu JD `job_type!=remote` thì `job_type` và `location` phải khớp exact giữa JD và CV.
- `seniority`: cả JD và CV đều phải có và phải khớp.
- `education` hard filter theo thứ bậc taxonomy: `unknown` < `high_school` < `bachelor` < `master` < `phd` (cả JD và CV đều phải có).
- Rule pass: education CV >= education JD.
- `required_certifications`: nếu JD flagged required thì CV phải có đầy đủ.

3. Scoring
- `title_sim`, `skills_sim`, `req_exp_sim`, `req_summary_sim`.
- `skills_sim = 0.6 * semantic + 0.4 * exact_overlap`.
- Semantic scores consume stored pgvector embeddings. Reasoning remains
  deterministic and rule-based.

4. Rerank and return
- apply bonus/penalty.
- sort theo final score, lấy top 10.
- trả response trực tiếp, không persist kết quả.

## Final Score

```
final_score =
  0.35 * title_sim +
  0.35 * skills_sim +
  0.20 * req_exp_sim +
  0.10 * req_summary_sim +
  bonus_exact_skill - penalty_missing_required
```

## Output Contract

Mỗi run trả:
- top 10 matches.
- `rank` cho từng match.
- score breakdown.
- rule-based reasoning.
- `runtime_ms_total` và runtime theo stage.

## Additive Hybrid Pipeline

Hybrid matching lives under `/api/v2/prototype/matching-hybrid/*` and runs in
parallel with the original matcher. It reuses the same PostgreSQL V2 rows and
embeddings, returns scores on a `0..100` scale, and does not alter the original
`/api/v2/prototype/matching/*` response contract.

Hybrid scoring groups:
- Calculated with current V2 schema: `title_score`, `skills_score`,
  `experience_score`, `seniority_score`, `education_score`,
  `certification_score`, `location_score`, `job_type_score`.
- Skipped because current V2 schema has no fields: `project_score`,
  `language_score`, `salary_score`.

Hybrid empty-field behavior:
- Both sides empty: skip the group and remove its weight from final scoring.
- JD has data but CV is empty: score the group as `0`, or fail strict required
  filters for location, seniority, education, certification, or job type.
- JD is empty and CV has data: skip the group and do not penalize the candidate.

Hybrid final score normalizes only valid group weights:

```
normalized_weight = group_weight / sum(valid_group_weights)
final_score = sum(group_score * normalized_weight)
```

If a hybrid semantic group has no MiniLM embedding available, it falls back to
deterministic text similarity for that group and returns a warning. It never
falls back to a remote embedding or LLM API.

## Out of Scope For Run-Only Prototype

- Persist/query/delete match results.
- Benchmark quality bằng labeled set.
- Auth/role guard riêng cho route prototype.
- Compare with pipelines outside the V2 prototype.
