## Draft Schema Reconciliation

The user-provided draft is a useful starting point for users, organizations,
profiles, resumes, jobs, embeddings, applications, and application events. The
canonical production docs require these adjustments:

- Add `uploaded_documents` and `parse_jobs` for original file tracking and async
  parsing.
- Add `recruiter_invites`, `notifications`, and `audit_logs`.
- Use canonical `applications.status` values:
  `submitted | shortlisted | rejected | hired | withdrawn`.
- Keep `candidate_resumes` as the CV-side matching entity and `job_posts` as the
  JD-side matching entity.
- Keep separate nullable `VECTOR(384)` embedding tables with mandatory
  `embedding_version`.
- Treat legacy prototype tables (`job_posts_v2`, `candidate_profiles_v2`,
  `job_embeddings_v2`, `candidate_embeddings_v2`) as non-production artifacts.

## Global Conventions

- Primary keys use `BIGINT GENERATED BY DEFAULT AS IDENTITY`.
- Timestamps use `TIMESTAMPTZ`.
- Enum-like values use `TEXT` or `VARCHAR` plus `CHECK` constraints, not
  PostgreSQL native enum types.
- Normalized skills and certifications use `TEXT[] NOT NULL DEFAULT '{}'`.
- Vector columns use pgvector `VECTOR(384)` and remain nullable so missing field
  embeddings can score `0` without failing a matching run.
- `created_at` and `updated_at` default to `now()` where both are present.
- Foreign keys default to restrictive behavior unless the lifecycle explicitly
  requires cascading child rows.

## Core Identity Tables

### `users`

```sql
CREATE TABLE users (
    user_id       BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email         VARCHAR(320) NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,
    role          VARCHAR(32) NOT NULL,
    status        VARCHAR(32) NOT NULL DEFAULT 'active',
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT now(),

    CONSTRAINT users_role_chk
        CHECK (role IN ('candidate', 'recruiter', 'admin')),
    CONSTRAINT users_status_chk
        CHECK (status IN ('active', 'invited', 'disabled'))
);

CREATE INDEX users_role_status_idx ON users (role, status);
```

Service-level invariants:

- Passwords are stored only as secure hashes.
- Disabled users cannot publish, activate, apply, invite, or run matching.
- Role-specific profile requirements are enforced in services.

### `organizations`

```sql
CREATE TABLE organizations (
    organization_id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name            VARCHAR(255) NOT NULL,
    slug            VARCHAR(255) UNIQUE,
    logo_url        TEXT,
    about           TEXT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Seed data:

- The production seed/migration must create exactly one predefined Independent
  organization row for recruiter onboarding option `Khác`.
- Recommended canonical values:
  - `name = 'Independent'`
  - `slug = 'independent'`
  - `logo_url = NULL`
- Frontend and services must discover this row by config/bootstrap or lookup;
  they must not hard-code an environment-specific `organization_id`.
- This seed row is a shared marketplace profile bucket, not tenant isolation.

### `recruiter_profiles`

```sql
CREATE TABLE recruiter_profiles (
    user_id         BIGINT PRIMARY KEY REFERENCES users(user_id),
    organization_id BIGINT NOT NULL REFERENCES organizations(organization_id),
    full_name       VARCHAR(255) NOT NULL,
    title           VARCHAR(255),
    phone           VARCHAR(64),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX recruiter_profiles_organization_id_idx
    ON recruiter_profiles (organization_id);
```

Service-level invariants:

- `user_id` must reference a `users.role = recruiter` account.
- Recruiter actions over jobs and invites must be scoped to jobs they own or are
  allowed to manage for their organization.

### `candidate_profiles`

```sql
CREATE TABLE candidate_profiles (
    user_id                BIGINT PRIMARY KEY REFERENCES users(user_id),
    full_name              VARCHAR(255) NOT NULL,
    phone                  VARCHAR(64),
    current_location       VARCHAR(64),
    total_experience_years INTEGER,
    headline               VARCHAR(255),
    created_at             TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at             TIMESTAMPTZ NOT NULL DEFAULT now(),

    CONSTRAINT candidate_profiles_current_location_chk
        CHECK (
            current_location IS NULL
            OR current_location IN ('ha_noi', 'tp_hcm', 'da_nang')
        ),
    CONSTRAINT candidate_profiles_total_experience_chk
        CHECK (
            total_experience_years IS NULL
            OR total_experience_years >= 0
        )
);
```

Service-level invariant:

- `user_id` must reference a `users.role = candidate` account.
- `total_experience_years` is display metadata only in MVP and is not a
  matching hard filter or score input.

## Marketplace Entity Tables

### `candidate_resumes`

```sql
CREATE TABLE candidate_resumes (
    resume_id         BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    candidate_user_id BIGINT NOT NULL REFERENCES users(user_id),
    title             TEXT NOT NULL,
    summary           TEXT NOT NULL DEFAULT '',
    experience        TEXT NOT NULL DEFAULT '',
    skills            TEXT[] NOT NULL DEFAULT '{}',
    location          VARCHAR(64) NOT NULL,
    job_type          VARCHAR(64) NOT NULL,
    seniority         VARCHAR(64) NOT NULL,
    education         VARCHAR(64) NOT NULL,
    certifications    TEXT[] NOT NULL DEFAULT '{}',
    is_primary        BOOLEAN NOT NULL DEFAULT false,
    status            VARCHAR(32) NOT NULL DEFAULT 'draft',
    created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at        TIMESTAMPTZ NOT NULL DEFAULT now(),

    CONSTRAINT candidate_resumes_status_chk
        CHECK (status IN ('draft', 'active', 'archived')),
    CONSTRAINT candidate_resumes_location_chk
        CHECK (location IN ('ha_noi', 'tp_hcm', 'da_nang')),
    CONSTRAINT candidate_resumes_job_type_chk
        CHECK (job_type IN ('remote', 'fulltime', 'parttime')),
    CONSTRAINT candidate_resumes_seniority_chk
        CHECK (seniority IN ('intern', 'fresher', 'junior', 'mid', 'senior', 'lead')),
    CONSTRAINT candidate_resumes_education_chk
        CHECK (education IN ('lop_9', 'lop_12', 'dai_hoc', 'thac_si', 'tien_si'))
);

CREATE INDEX candidate_resumes_candidate_user_id_idx
    ON candidate_resumes (candidate_user_id);
CREATE INDEX candidate_resumes_candidate_status_idx
    ON candidate_resumes (candidate_user_id, status);
CREATE INDEX candidate_resumes_public_pool_idx
    ON candidate_resumes (status, location, job_type, seniority, education)
    WHERE status = 'active';
CREATE INDEX candidate_resumes_skills_gin_idx
    ON candidate_resumes USING GIN (skills);
CREATE INDEX candidate_resumes_certifications_gin_idx
    ON candidate_resumes USING GIN (certifications);
```

Service-level invariants:

- `candidate_user_id` must reference a candidate user.
- A candidate may have multiple active resumes.
- `is_primary` is a display/default-selection hint only; `status = active`
  controls public pool visibility.

```sql
CREATE TABLE job_posts (

    updated_at              TIMESTAMPTZ NOT NULL DEFAULT now(),

        CHECK (expires_at IS NULL OR published_at IS NULL OR expires_at > published_at)

    ON job_posts USING GIN (required_certifications);
```

Service-level invariants:

- `recruiter_user_id` must reference a recruiter user.
- `published_at` is set when status first becomes `published`.
- Closed jobs cannot accept new applications or invites.
- Published jobs control public job search and resume -> job matching
  visibility.

## Documents And Parse Jobs

### `uploaded_documents`

```sql
CREATE TABLE uploaded_documents (
    document_id        BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    owner_user_id      BIGINT NOT NULL REFERENCES users(user_id),
    document_type      VARCHAR(64) NOT NULL,
    object_key         TEXT,
    file_url           TEXT,
    original_filename  TEXT NOT NULL,
    mime_type          VARCHAR(255) NOT NULL,
    file_size_bytes    BIGINT NOT NULL,
    resume_id          BIGINT REFERENCES candidate_resumes(resume_id),
    job_id             BIGINT REFERENCES job_posts(job_id),
    created_at         TIMESTAMPTZ NOT NULL DEFAULT now(),

    CONSTRAINT uploaded_documents_type_chk
        CHECK (document_type IN ('candidate_resume', 'job_post')),
    CONSTRAINT uploaded_documents_storage_ref_chk
        CHECK (object_key IS NOT NULL OR file_url IS NOT NULL),
    CONSTRAINT uploaded_documents_size_chk
        CHECK (file_size_bytes > 0),
    CONSTRAINT uploaded_documents_target_chk
        CHECK (
            (document_type = 'candidate_resume' AND job_id IS NULL)
            OR
            (document_type = 'job_post' AND resume_id IS NULL)
        )
);

CREATE INDEX uploaded_documents_owner_created_idx
    ON uploaded_documents (owner_user_id, created_at DESC);
CREATE INDEX uploaded_documents_resume_id_idx ON uploaded_documents (resume_id);
CREATE INDEX uploaded_documents_job_id_idx ON uploaded_documents (job_id);
```

Service-level invariants:

- Original files live in object storage; the DB stores metadata and object
  references.
- Parse failure must not delete the original file.
- `resume_id` or `job_id` may be null while a business entity is being created,
  but once linked it must match `document_type`.

### `parse_jobs`

```sql
CREATE TABLE parse_jobs (
    parse_job_id                BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    document_id                 BIGINT NOT NULL REFERENCES uploaded_documents(document_id),
    target_entity_type          VARCHAR(64) NOT NULL,
    resume_id                   BIGINT REFERENCES candidate_resumes(resume_id),
    job_id                      BIGINT REFERENCES job_posts(job_id),
    status                      VARCHAR(32) NOT NULL DEFAULT 'queued',
    extracted_text              TEXT,
    extracted_text_object_key   TEXT,
    parser_version              VARCHAR(128) NOT NULL,
    embedding_version_requested VARCHAR(128) NOT NULL,
    error_code                  VARCHAR(128),
    error_message               TEXT,
    started_at                  TIMESTAMPTZ,
    finished_at                 TIMESTAMPTZ,
    created_at                  TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at                  TIMESTAMPTZ NOT NULL DEFAULT now(),

    CONSTRAINT parse_jobs_target_type_chk
        CHECK (target_entity_type IN ('candidate_resume', 'job_post')),
    CONSTRAINT parse_jobs_status_chk
        CHECK (status IN ('queued', 'processing', 'succeeded', 'failed')),
    CONSTRAINT parse_jobs_target_chk
        CHECK (
            (
                target_entity_type = 'candidate_resume'
                AND job_id IS NULL
                AND (resume_id IS NOT NULL OR status IN ('queued', 'processing', 'failed'))
            )
            OR
            (
                target_entity_type = 'job_post'
                AND resume_id IS NULL
                AND (job_id IS NOT NULL OR status IN ('queued', 'processing', 'failed'))
            )
        ),
    CONSTRAINT parse_jobs_text_ref_chk
        CHECK (extracted_text IS NOT NULL OR extracted_text_object_key IS NOT NULL OR status IN ('queued', 'processing', 'failed')),
    CONSTRAINT parse_jobs_finished_after_started_chk
        CHECK (finished_at IS NULL OR started_at IS NULL OR finished_at >= started_at)
);

CREATE INDEX parse_jobs_document_id_idx ON parse_jobs (document_id);
CREATE INDEX parse_jobs_status_created_idx ON parse_jobs (status, created_at);
CREATE INDEX parse_jobs_resume_id_idx ON parse_jobs (resume_id);
CREATE INDEX parse_jobs_job_id_idx ON parse_jobs (job_id);
```

Service-level invariants:

- Upload returns after file persistence and parse job creation.
- A queued or processing parse job may not yet have `resume_id` or `job_id`
  because the worker has not produced the structured target entity. A succeeded
  parse job must link exactly one target entity matching `target_entity_type`.
- A failed parse job may remain unlinked when extraction or structured parsing
  fails before entity creation.
- Worker transitions status through `queued`, `processing`, and then
  `succeeded` or `failed`.
- Failed parse jobs create user-visible notification and audit entries.
- Raw file contents, passwords, and JWTs must not be written to operational logs.

## Applications And Invites

### `applications`

```sql
CREATE TABLE applications (
    application_id    BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    job_id            BIGINT NOT NULL REFERENCES job_posts(job_id),
    candidate_user_id BIGINT NOT NULL REFERENCES users(user_id),
    resume_id         BIGINT NOT NULL REFERENCES candidate_resumes(resume_id),
    status            VARCHAR(32) NOT NULL DEFAULT 'submitted',
    applied_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at        TIMESTAMPTZ NOT NULL DEFAULT now(),

    CONSTRAINT applications_status_chk
        CHECK (status IN ('submitted', 'shortlisted', 'rejected', 'hired', 'withdrawn')),
    CONSTRAINT applications_job_resume_uniq
        UNIQUE (job_id, resume_id)
);

CREATE INDEX applications_job_status_idx ON applications (job_id, status);
CREATE INDEX applications_candidate_status_idx
    ON applications (candidate_user_id, status);
CREATE INDEX applications_resume_id_idx ON applications (resume_id);
```

Service-level invariants:

- Matching results never create applications.
- Candidate apply creates an application with `submitted`.
- Accepted recruiter invite creates an application if no `(job_id, resume_id)`
  row exists.
- Closed jobs do not accept new applications.
- `candidate_user_id` must own `resume_id`.

```sql
            OR from_status IN ('submitted', 'shortlisted', 'rejected', 'hired', 'withdrawn')

    CONSTRAINT application_events_to_status_chk

    ON application_events (application_id, created_at);
CREATE INDEX application_events_actor_created_idx
    ON application_events (actor_user_id, created_at DESC);
```

Service-level invariant:

- `application_events` is append-only status history. Updates and deletes should
  be blocked by repository/service policy unless an explicit admin correction
  workflow is later specified.

Service-level invariants:

- Pending invites do not create applications.
- Accepted invites create applications when no duplicate application exists.
- Rejected invites create no application.
- `candidate_user_id` must own `resume_id`.
- `recruiter_user_id` must own or be authorized for `job_id`.

Notification types are application-defined strings. Required MVP event families:

- application submitted.
- recruiter invite sent.
- invite accepted or rejected.

Service-level invariant:

- Email delivery failure must be logged and must not roll back the business
  transaction.

## Relationship Summary

- `users` is the root identity table.
- `candidate_profiles.user_id`, `recruiter_profiles.user_id`, resume owners,
  job recruiters, application candidates, invite actors, notification
  recipients, and audit actors all reference `users`.
- `organizations` owns recruiter organization profile data and is referenced by
  recruiter profiles and job posts.
- `candidate_resumes` and `job_posts` are the matching entities.
- `uploaded_documents` references the original file owner and optionally links to
  the resume or job created from that file.
- `parse_jobs` references `uploaded_documents`; succeeded parse jobs link exactly
  one target entity, while queued, processing, or failed jobs may remain
  unlinked until structured entity creation succeeds.
- Embedding tables are one-to-one child tables of resumes and jobs.
- `applications` links one job to one candidate resume and is unique by
  `(job_id, resume_id)`.
- `application_events` is append-only history for application status changes.
- `recruiter_invites` links one recruiter-owned job to one active resume and may
  create an application on accept.
- `notifications` and `audit_logs` record lifecycle side effects without
  controlling business state.

## Index Strategy

- Owner dashboards: `(owner_user_id, created_at DESC)`, `(candidate_user_id,
  status)`, `(recruiter_user_id, status)`, and organization/recruiter foreign key
  indexes.
- Public pool queries: partial indexes for `candidate_resumes.status = active`
  and `job_posts.status = published`, including matching hard-filter fields.
- Application lifecycle: `(job_id, status)`, `(candidate_user_id, status)`, and
  application event history by `(application_id, created_at)`.
- Invite lifecycle: `(job_id, status)`, `(candidate_user_id, status)`, and
  `(recruiter_user_id, status)`.
- Array filters: GIN indexes on normalized skills and certification arrays.
- Semantic retrieval: pgvector HNSW indexes on embedding columns after
  production volume and model settings are confirmed.
- Audit/operations: event, actor, target, and JSONB metadata indexes support
  admin monitoring and incident review.

## Deferred Review Notes

The following review notes are intentionally non-blocking for this LLD:

- `docs/REQUIREMENTS.md` mentions login/auth security audit events "where
  useful". This LLD keeps `audit_logs` generic enough to store those events, but
  does not require a dedicated auth audit table or a fixed auth event taxonomy.
- Matching request count and latency are observability signals. They may be
  captured by metrics/logging infrastructure rather than PostgreSQL tables. Add
  a `matching_runs` or similar table only if admin product requirements later
  need persisted, queryable matching history.
- Requirement wording may refer to job "unpublish". For this schema, the job
  lifecycle remains `draft | published | closed`; unpublish-like behavior maps
  to leaving `published`, normally by setting `status = closed`, unless a later
  requirement introduces a separate state.

## Non-Goals For This LLD

- This document does not define an Alembic migration.
- This document does not define ORM models or repository APIs.
- This document does not preserve or restore the legacy V2 prototype schema.
- This document does not add tenant/company isolation. MVP uses a shared public
  marketplace pool controlled by resume/job visibility statuses.
